# commit-on-feature（会话型 session）

## 硬性规则（每轮必须执行）

**凡是本轮产生了改动、执行了 git 命令、或用户说了"提交"/"commit"/"推送"/"push" 的回复**，
必须：

1. 判断本轮是否满足以下触发条件。
2. 如果满足 → 回复前缀加上 `[Trigger: commit-on-feature]`，并按下方步骤执行。
3. 如果不满足 → 简要说明原因（如"不在功能分支"、"没有改动"）。

**不要让用户主动说"提交"。** 如果你在本轮做了改动，就自动检查、自动提交。

## 触发条件

满足以下 **任意一条** 即触发：
- 本轮修改/创建了文件，正准备回复。
- 用户说了"提交"/"commit"/"推送"/"push"/"更新"/"提交到 git"。
- `git status` 显示有暂存或未暂存的改动且属于当前工作。

## 干净前置检查（任一不满足→不自动做，只提醒用户）

1. 在功能分支上（非 `main`/`master`/`release*`，非 detached HEAD）。
2. 不在 merge/rebase/cherry-pick 中途。
3. 改动仅包含本轮的改动（未混入无关改动）。
4. 无密钥/大文件（`.env`、`credentials.*`、模型权重、数据集等）。

## 触发后（开头标来源 `[Trigger: commit-on-feature]`）

- **小改动**（单文件修复、配置调整）：`git add -A` → `git commit`（不推送）。
- **大改动**（多文件里程碑、用户可见变更）：提交后 `git push`（当前分支）。

不确定时按小改动处理。一个逻辑改动一个提交，不做半成品 "WIP" 提交。

## 提交后确认

1. 用 `git log -1 --oneline` 确认提交已创建。
2. 如推送到远端，确认推送成功（未被拒绝、无冲突）。
3. 汇报摘要：改动范围、是否推送、目标分支。

## 红线（绝不越界）

- 禁止 `git push --force` 或 `--force-with-lease`。
- 禁止修改已发布的提交（amend published commits）。
- 禁止在此触发器内切换分支或 rebase。
- 推送失败时（无 upstream、冲突、认证问题）：停止并报告，不以 force 重试。
